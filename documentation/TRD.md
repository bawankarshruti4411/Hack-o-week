## 1. System Architecture
The application is built on the **MVC (Model-View-Controller)** pattern, decoupled into routes, controllers, services, and data models to enforce clean separation of concerns.

```
                  ┌──────────────────────┐
                  │     Client Request   │
                  └──────────┬───────────┘
                             │
                             ▼
                  ┌──────────────────────┐
                  │  Express Middleware  │ (Helmet, CORS, Rate Limit, Parsing)
                  └──────────┬───────────┘
                             │
                             ▼
                  ┌──────────────────────┐
                  │    Router Routing    │ (auth.routes, student.routes)
                  └──────────┬───────────┘
                             │
                             ▼
                  ┌──────────────────────┐
                  │  Request Validator   │ (express-validator)
                  └──────────┬───────────┘
                             │
                             ▼
                  ┌──────────────────────┐
                  │      Controller      │ (Extracts inputs, invokes services)
                  └──────────┬───────────┘
                             │
                             ▼
                  ┌──────────────────────┐
                  │       Services       │ (Executes core business logic)
                  └──────────┬───────────┘
                             │
                             ▼
                  ┌──────────────────────┐
                  │     Prisma ORM       │ (Generates SQL queries dynamically)
                  └──────────┬───────────┘
                             │
                             ▼
                  ┌──────────────────────┐
                  │  PostgreSQL Database │
                  └──────────────────────┘
```

## 2. Technology Stack

| Layer | Component | Selection | Justification |
| :--- | :--- | :--- | :--- |
| **Runtime** | Engine | Node.js (v18+) | Event-driven, asynchronous I/O model well-suited for high-throughput REST APIs. |
| **Framework** | Web Server | Express.js | Lightweight and unopinionated routing engine with robust middleware support. |
| **ORM** | Data Access | Prisma ORM | Autogenerated typesafe queries, easy schema declarations, and declarative migrations. |
| **Database** | Storage | PostgreSQL | Relational storage ensuring ACID compliance, relational integrity (FKs), and robust JSON query features. |
| **Auth** | Security | JWT + bcryptjs | Token-based stateless authentication. Bcrypt provides secure, salt-hashed credentials. |
| **Validator**| Input Sanitation| express-validator | Declarative validator based on validator.js to prevent SQL injection and schema corruption. |

## 3. Database Schema Design (Prisma)

The application utilizes two primary relations: `User` and `Student`. A user has a **one-to-many** relationship with students (one admin user can register/track multiple students).

### 3.1 Model definitions (`schema.prisma`)

```prisma
model User {
  id        String    @id
  name      String
  email     String    @unique
  password  String
  role      String    @default("student")
  isActive  Boolean   @default(true)
  lastLogin DateTime?
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  students  Student[] @relation("CreatedStudents")
}

model Student {
  id          String   @id
  name        String
  email       String   @unique
  phone       String
  age         Int
  department  String
  semester    Int
  address     Json     @default("{}")
  createdBy   String?
  creator     User?    @relation("CreatedStudents", fields: [createdBy], references: [id], onDelete: SetNull)
  isActive    Boolean  @default(true)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}
```

## 5. Security & Middleware Configuration

### 5.1 Helmet Middleware
Secures HTTP response headers:
- Disables `x-powered-by` to prevent framework identification.
- Sets `X-Frame-Options: SAMEORIGIN` to mitigate Clickjacking.
- Sets strict `Content-Security-Policy`.

### 5.2 CORS Configuration
Allows browser-level requests. Dynamically matches localhost environments in development:
```javascript
const isLocalhost = /^https?:\/\/(localhost|127\.0\.0\.1)(:\d+)?$/.test(origin);
```

### 5.3 Authentication Guard (`protect`)
Verifies Bearer JWTs:
1. Extracts token from `Authorization` header (`Bearer <token>`).
2. Calls `jwt.verify` against `JWT_SECRET`.
3. Verifies user exists in PostgreSQL and is active (`isActive === true`).
4. Attaches formatted user record (`_id`, `role`, etc.) to `req.user`.

### 5.4 Input Validation
Validation is handled via `express-validator` rules. If errors exist, they are compiled in `validate.middleware.js` and returned with code `422 Unprocessable Entity` listing the fields and error descriptions.
